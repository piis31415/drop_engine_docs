xmldom is a javascript \href{https://ponyfill.com/}{\texttt{ ponyfill}} to provide the following APIs \href{https://caniuse.com/xml-serializer}{\texttt{ that are present in modern browsers}} to other runtimes\+:
\begin{DoxyItemize}
\item convert an XML string into a DOM tree 
\begin{DoxyCode}{0}
\DoxyCodeLine{new\ DOMParser().parseFromString(xml,\ mimeType)\ =>\ Document}

\end{DoxyCode}

\item create, access and modify a DOM tree 
\begin{DoxyCode}{0}
\DoxyCodeLine{new\ DOMImplementation().createDocument(...)\ =>\ Document}

\end{DoxyCode}

\item serialize a DOM tree back into an XML string 
\begin{DoxyCode}{0}
\DoxyCodeLine{new\ XMLSerializer().serializeToString(node)\ =>\ string}

\end{DoxyCode}

\end{DoxyItemize}

The target runtimes {\ttfamily xmldom} supports are currently Node \texorpdfstring{$>$}{>}= v10 (ES5) and Rhino (\href{https://github.com/xmldom/xmldom/discussions/214}{\texttt{ not tested as part of CI}}).

When deciding how to fix bugs or implement features, {\ttfamily xmldom} tries to stay as close as possible to the various related specifications/standards. ~\newline
 As indicated by the version starting with {\ttfamily 0.}, this implementation is not feature complete and some implemented features differ from what the specifications describe. ~\newline
 {\bfseries{Issues and PRs for such differences are always welcome, even when they only provide a failing test case.}}

The implementation is based on several specifications\+:

\href{https://raw.githubusercontent.com/xmldom/xmldom/master/docs/specs.svg}{\texttt{ }}\hypertarget{md__2_users_2hello_2_documents_2_git_hub_2finalproject-engine-drop-table-engines_2_engine_2src_2b04a944b5390ac595140a588eda936f4_autotoc_md1791}{}\doxysubsection{\texorpdfstring{DOM Parsing and Serialization}{DOM Parsing and Serialization}}\label{md__2_users_2hello_2_documents_2_git_hub_2finalproject-engine-drop-table-engines_2_engine_2src_2b04a944b5390ac595140a588eda936f4_autotoc_md1791}
From the \href{https://www.w3.org/TR/2016/WD-DOM-Parsing-20160517/}{\texttt{ W3C DOM Parsing and Serialization (WD 2016)}} {\ttfamily xmldom} provides an implementation for the interfaces\+:
\begin{DoxyItemize}
\item {\ttfamily DOMParser}
\item {\ttfamily XMLSerializer}
\end{DoxyItemize}

Note that there are some known deviations between this implementation and the W3 specifications.

Note\+: \href{https://w3c.github.io/DOM-Parsing/}{\texttt{ The latest version of this spec}} has the status "{}\+Editors Draft"{}, since it is under active development. One major change is that \href{https://w3c.github.io/DOM-Parsing/\#the-domparser-interface}{\texttt{ the definition of the {\ttfamily DOMParser} interface has been moved to the HTML spec}}\hypertarget{md__2_users_2hello_2_documents_2_git_hub_2finalproject-engine-drop-table-engines_2_engine_2src_2b04a944b5390ac595140a588eda936f4_autotoc_md1792}{}\doxysubsection{\texorpdfstring{DOM}{DOM}}\label{md__2_users_2hello_2_documents_2_git_hub_2finalproject-engine-drop-table-engines_2_engine_2src_2b04a944b5390ac595140a588eda936f4_autotoc_md1792}
The original author claims that xmldom implements \mbox{[}DOM Level 2\mbox{]} in a "{}fully compatible"{} way and some parts of \mbox{[}DOM Level 3\mbox{]}, but there are not enough tests to prove this. Both Specifications are now superseded by the \mbox{[}DOM Level 4 aka Living standard\mbox{]} wich has a much broader scope than xmldom.
