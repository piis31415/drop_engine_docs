The {\bfseries{node-\/addon-\/api}} team loves contributions. There are many ways in which you can contribute to {\bfseries{node-\/addon-\/api}}\+:
\begin{DoxyItemize}
\item Source code fixes
\item Additional tests
\item Documentation improvements
\item Joining the N-\/\+API working group and participating in meetings
\end{DoxyItemize}\hypertarget{md__2_users_2hello_2_documents_2_git_hub_2finalproject-engine-drop-table-engines_2_engine_2src_227a8b0eb3ee755d397f31b953ef7e080_autotoc_md7606}{}\doxysection{\texorpdfstring{Source changes}{Source changes}}\label{md__2_users_2hello_2_documents_2_git_hub_2finalproject-engine-drop-table-engines_2_engine_2src_227a8b0eb3ee755d397f31b953ef7e080_autotoc_md7606}
{\bfseries{node-\/addon-\/api}} is meant to be a thin convenience wrapper around N-\/\+API. With this in mind, contributions of any new APIs that wrap around a core N-\/\+API API will be considered for merge. However, changes that wrap existing {\bfseries{node-\/addon-\/api}} APIs are encouraged to instead be provided as an ecosystem module. The {\bfseries{node-\/addon-\/api}} team is happy to link to a curated set of modules that build on top of {\bfseries{node-\/addon-\/api}} if they have broad usefulness to the community and promote a recommended idiom or pattern.\hypertarget{md__2_users_2hello_2_documents_2_git_hub_2finalproject-engine-drop-table-engines_2_engine_2src_227a8b0eb3ee755d397f31b953ef7e080_autotoc_md7607}{}\doxysubsection{\texorpdfstring{Rationale}{Rationale}}\label{md__2_users_2hello_2_documents_2_git_hub_2finalproject-engine-drop-table-engines_2_engine_2src_227a8b0eb3ee755d397f31b953ef7e080_autotoc_md7607}
The N-\/\+API team considered a couple different approaches with regards to changes extending {\bfseries{node-\/addon-\/api}}
\begin{DoxyItemize}
\item Larger core module -\/ Incorporate these helpers and patterns into {\bfseries{node-\/addon-\/api}}
\item Extras package -\/ Create a new package (strawman name \textquotesingle{}{\bfseries{node-\/addon-\/api}}-\/extras\textquotesingle{}) that contain utility classes and methods that help promote good patterns and idioms while writing native addons with {\bfseries{node-\/addon-\/api}}.
\item Ecosystem -\/ Encourage creation of a module ecosystem around {\bfseries{node-\/addon-\/api}} where folks can build on top of it.
\end{DoxyItemize}\hypertarget{md__2_users_2hello_2_documents_2_git_hub_2finalproject-engine-drop-table-engines_2_engine_2src_227a8b0eb3ee755d397f31b953ef7e080_autotoc_md7608}{}\doxysubsubsection{\texorpdfstring{Larger Core}{Larger Core}}\label{md__2_users_2hello_2_documents_2_git_hub_2finalproject-engine-drop-table-engines_2_engine_2src_227a8b0eb3ee755d397f31b953ef7e080_autotoc_md7608}
This is probably our simplest option in terms of immediate action needed. It would involve landing any open PRs against {\bfseries{node-\/addon-\/api}}, and continuing to encourage folks to make PRs for utility helpers against the same repository.

The downside of the approach is the following\+:
\begin{DoxyItemize}
\item Less coherency for our API set
\item More maintenance burden on the N-\/\+API WG core team.
\end{DoxyItemize}\hypertarget{md__2_users_2hello_2_documents_2_git_hub_2finalproject-engine-drop-table-engines_2_engine_2src_227a8b0eb3ee755d397f31b953ef7e080_autotoc_md7609}{}\doxysubsubsection{\texorpdfstring{Extras Package}{Extras Package}}\label{md__2_users_2hello_2_documents_2_git_hub_2finalproject-engine-drop-table-engines_2_engine_2src_227a8b0eb3ee755d397f31b953ef7e080_autotoc_md7609}
This involves us spinning up a new package which contains the utility classes and methods. This has the benefit of having a separate module where helpers which make it easier to implement certain patterns and idioms for native addons easier.

The downside of this approach is the following\+:
\begin{DoxyItemize}
\item Potential for confusion -\/ we\textquotesingle{}ll need to provide clear documentation to help the community understand where a particular contribution should be directed to (what belongs in {\bfseries{node-\/addon-\/api}} vs {\bfseries{node-\/addon-\/api-\/extras}})
\item Need to define the level of support/\+API guarantees
\item Unclear if the maintenance burden on the N-\/\+API WG is reduced or not
\end{DoxyItemize}\hypertarget{md__2_users_2hello_2_documents_2_git_hub_2finalproject-engine-drop-table-engines_2_engine_2src_227a8b0eb3ee755d397f31b953ef7e080_autotoc_md7610}{}\doxysubsubsection{\texorpdfstring{Ecosystem}{Ecosystem}}\label{md__2_users_2hello_2_documents_2_git_hub_2finalproject-engine-drop-table-engines_2_engine_2src_227a8b0eb3ee755d397f31b953ef7e080_autotoc_md7610}
This doesn\textquotesingle{}t require a ton of up-\/front work from the N-\/\+API WG. Instead of accepting utility PRs into {\bfseries{node-\/addon-\/api}} or creating and maintaining a new module, the WG will encourage the creation of an ecosystem of modules that build on top of {\bfseries{node-\/addon-\/api}}, and provide some level of advertising for these modules (listing them out on the repository/wiki, using them in workshops/tutorials etc).

The downside of this approach is the following\+:
\begin{DoxyItemize}
\item Potential for lack of visibility -\/ evangelism and education is hard, and module authors might not find right patterns and instead implement things themselves
\item There might be greater friction for the N-\/\+API WG in evolving APIs since the ecosystem would have taken dependencies on the API shape of {\bfseries{node-\/addon-\/api}} 
\end{DoxyItemize}
