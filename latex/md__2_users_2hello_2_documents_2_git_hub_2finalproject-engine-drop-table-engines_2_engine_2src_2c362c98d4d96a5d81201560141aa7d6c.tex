Sometimes you feel the itch to open {\itshape all the files} at once. You want to pass a bunch of streams around, so the consumer does not need to worry where the data comes from. From a software design point-\/of-\/view this sounds entirely reasonable. Then there is that neat little function {\ttfamily fs.\+create\+Read\+Stream()} that opens a file and gives you a nice {\ttfamily fs.\+Read\+Stream} to pass around, so you use what the mighty creator deities of node bestowed upon you.

\begin{quote}
{\ttfamily Error\+: EMFILE, too many open files} ~\newline
 ─ {\itshape node} \end{quote}
This package provides two classes based on the node\textquotesingle{}s Streams3 API (courtesy of {\ttfamily readable-\/stream} to ensure a stable version).\hypertarget{md__2_users_2hello_2_documents_2_git_hub_2finalproject-engine-drop-table-engines_2_engine_2src_2c362c98d4d96a5d81201560141aa7d6c_autotoc_md6656}{}\doxysection{\texorpdfstring{Class\+: lazystream.\+Readable}{Class\+: lazystream.\+Readable}}\label{md__2_users_2hello_2_documents_2_git_hub_2finalproject-engine-drop-table-engines_2_engine_2src_2c362c98d4d96a5d81201560141aa7d6c_autotoc_md6656}

\item {\ttfamily options} \texorpdfstring{$\ast$}{*}\{Object\}\texorpdfstring{$\ast$}{*} ~\newline
 Options for the underlying {\ttfamily Pass\+Through} stream, accessible by {\ttfamily fn}.

Creates a new readable stream. Once the stream is accessed (for example when you call its {\ttfamily read()} method, or attach a {\ttfamily data}-\/event listener) the {\ttfamily fn} function is called with the outer {\ttfamily lazystream.\+Readable} instance bound to {\ttfamily this}.

If you pass an {\ttfamily options} object to the constuctor, you can access it in your {\ttfamily fn} function.
